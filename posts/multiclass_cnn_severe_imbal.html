‚ú® Introduction / TL;DR

I took on the challenge of building a multi-class Convolutional Neural Network from scratch using PyTorch ‚Äî with a twist. Merged, this totals 1,804 images across 18 classes with heavy imbalance: 1 class dominates half the dataset. Everything is now preprocessed locally and trained fully from scratch.

üß© Issue #1: Merging Datasets Broke Class Indexing

‚ùå Problem:

This project involved merging two datasets:

  • INRIA: One class (person) with train/, valid/, test/ splits
  • Animal Dataset: 17 animal classes (e.g., badger, zebra, etc.) with the same structure

PyTorch‚Äôs ImageFolder assigns indices alphabetically, so both datasets started at 0. When combined using ConcatDataset, this caused:

  • Label collisions (e.g., person and badger both mapped to index 0)
  • Misclassified training examples
  • Broken evaluation logic

üí° Fix:

I applied a label offset before combining the datasets:

offset = len(inria_classes)
animal_label = original_animal_label + offset

‚úîÔ∏è This made the label space consistent and eliminated overlap.

üßπ Issue #2: Empty or Mismatched Class Folders

‚ùå Problem:

  • Some folders were completely empty
  • Others had mismatched folder names vs. class labels
  • Missing images caused underrepresented classes

üí° Fix:

I built a dataset structure verifier that checked folder contents per split:

‚ï∞‚ï¥ 5. üßÆ Mean & Std Normalization: Precision, Consistency, and Caching

Normalizing image datasets using channel-wise mean and standard deviation is a critical step for stable and efficient training ‚Äî especially for CNNs. In my original capstone, I implemented a simple calculate_mean_std() function, but it had some rough edges. By the time I reached the final iteration, I had reworked it entirely for both accuracy and reusability.

Capstone Version: Single-Pass, Underestimated Variance
def calculate_mean_std(loader):
    mean = 0.
    std = 0.
    total_images = 0

    for data, _ in loader:
        batch_samples = data.size(0)
        total_images += batch_samples
        data = data.view(batch_samples, data.size(1), -1)
        mean += data.mean(2).sum(0)
        std += data.std(2).sum(0)

    mean /= total_images
    std /= total_images
    return mean, std

This version loops once, but underestimates variance due to image-based averaging.


Calculated Mean: tensor([0.4713, 0.4595, 0.4212])
Calculated Std:  tensor([0.2065, 0.2032, 0.2050])
Final Version: Double-Pass for True Pixel-Wise Accuracy
def compute_mean_std(loader):
    n_channels = 3
    mean = torch.zeros(n_channels)
    std = torch.zeros(n_channels)
    n_pixels = 0

    for data, _ in loader:
        batch_samples = data.size(0)
        data = data.view(batch_samples, data.size(1), -1)
        mean += data.sum(2).sum(0)
        n_pixels += data.size(2) * batch_samples

    mean /= n_pixels

    for data, _ in loader:
        batch_samples = data.size(0)
        data = data.view(batch_samples, data.size(1), -1)
        std += ((data - mean.view(1, -1, 1)) ** 2).sum(2).sum(0)

    std = torch.sqrt(std / n_pixels)
    return mean, std

This version does a full two-pass mean & std with pixel-based precision.


Computed Mean: tensor([0.4712, 0.4595, 0.4211])
Computed Std:  tensor([0.2522, 0.2475, 0.2572])
üíæ Added Caching
def get_or_compute_mean_std(loader, save_path='mean_std.pt'):
    if os.path.exists(save_path):
        stats = torch.load(save_path)
        print("Load mean/std via file.")
        return stats['mean'], stats['std'], "loaded"
    else:
        mean, std = compute_mean_std(loader)
        torch.save({'mean': mean, 'std': std}, save_path)
        print("Computed & saved mean/std.")
        return mean, std, "computed"

This function automatically loads mean_std.pt if available ‚Äî or computes it and caches the result. This caching logic reduces runtime and prevents redundant computation during repeated training sessions.

üìå Used later in: Transforms & Loaders cell ‚Äî ensuring that the normalization applied to the dataset matches across training, validation, and testing every time.

‚ï∞‚ï¥ 7. üßÆ Comparing Class Weighting Strategies

Another huge improvement I made to the pipeline was changing how I calculated class weights. I considered testing an approach called Class-Balanced Loss Based on Effective Number of Samples , introduced in a research paper by Yin Cui, Menglin Jia, Tsung-Yi Lin, Yang Song, and Serge Belongie. It was specifically developed to address dataset imbalance challenges. However, I ultimately opted for the more straightforward Log-Scaled Inverse Frequency method, replacing my original use of sklearn‚Äôs Balanced Class Weights. The difference in the resulting class weights (see below) was substantial.

üì¶ Reminde - Here‚Äôs what the data looked like after merging:

  • train/ ‚Üí person: 632 images | each animal class: ~37‚Äì38
  • valid/ ‚Üí person: 180 images | each animal class: ~10‚Äì11
  • test/ ‚Üí person: 90 images | each animal class: ~5‚Äì6

This created a severely imbalanced dataset where person made up ~50% of the total images. This imbalance was intentional to simulate realism but required careful loss weighting to prevent the model from ignoring minority classes.

In my capstone, I used sklearn‚Äôs compute_class_weight('balanced'). In practice, this method overcompensated:

üß™ Capstone Strategy: Balanced Weights

Formula:

w_c = (N) / (C √ó n_c)
  • w_c: weight for class c
  • N: total number of samples
  • C: number of classes
  • n_c: samples in class c

Used compute_class_weight('balanced') ‚Äî but it overcompensated:

üß† Final Strategy: Log-Scaled Weights

Formula:

w_c = log(1 + N / n_c)
  • w_c: weight for class c
  • N: total samples
  • n_c: samples in class c

‚ï∞‚ï¥ 8. üß± CNN Architecture Evolution & Mixed Precision Fixes

When it came time to define the actual model architecture, I iterated on a custom convolutional neural network built entirely from scratch using PyTorch‚Äôs nn.Sequential. üîß Capstone Version: Basic But Brittle

The original model got the job done but left room for improvement. Notable limitations included:

  • ‚ùå Static assignment of out_classes
  • ‚ùå Only one Dropout(p=0.2) layer ‚Äî not enough regularization
  • ‚ùå Vulnerable to mixed precision errors (e.g., float16 tensors passed directly to nn.Linear)
  • ‚úÖ Did correctly use AdaptiveAvgPool2d((1,1)) for flexible spatial size reduction

üêõ Bug: CUBLAS_STATUS_NOT_SUPPORTED

While enabling mixed precision (autocast()), the model initially crashed with a low-level CUDA error:

RuntimeError: CUDA error: CUBLAS_STATUS_NOT_SUPPORTED when calling cublasLtMatmulAlgoGetHeuristic

Root cause:
Passing a float16 tensor into an nn.Linear layer without manually restoring it to float32 precision ‚Äî a common issue on consumer GPUs or certain CUDA driver versions.

üí° The Fix: Insert a Lightweight Float32 Patch

Since my model used nn.Sequential (no manual forward() method), I couldn't just write x = x.float() inline.

Instead, I created a simple layer:

class ToFloat32(nn.Module):
    def forward(self, x):
        return x.float()

Inserted just after flattening:

nn.Flatten(),
ToFloat32(),  # üî• Converts float16 ‚Üí float32
nn.Linear(512, 512),

‚ï∞‚ï¥ 9. üß† Weight Initialization: Same Foundation, Still Solid

While many parts of the architecture evolved in the final version, my approach to weight initialization stayed consistent. I used a hybrid strategy tailored to the type of layer:

  • ‚úÖ Xavier Initialization for all fully connected (nn.Linear) layers ‚Äî balances variance across layers and supports stable convergence.
  • ‚úÖ Kaiming (He) Initialization for convolutional layers (nn.Conv2d) ‚Äî optimized for ReLU-like activations and preserves forward signal strength.
Code: Custom Weight Initialization Function 
def weights_init(m):
    if isinstance(m, nn.Linear):
        nn.init.xavier_uniform_(m.weight)
        if m.bias is not None:
            nn.init.zeros_(m.bias)
    elif isinstance(m, nn.Conv2d):
        nn.init.kaiming_normal_(m.weight, nonlinearity='relu')
        if m.bias is not None:
            nn.init.zeros_(m.bias)

Although I experimented with alternate activation functions like GELU, I kept nonlinearity='relu' for Kaiming as a safe and compatible default.

üß† Insight: This function was called right after the model was defined using net.apply(weights_init) ‚Äî ensuring reproducible and properly scaled initialization for each run.

‚öôÔ∏è Optimizer Upgrades: Smarter Weight Decay

Applied weight decay to all parameters ‚Äî including bias and batchnorm layers (suboptimal).

‚úÖ Final Version

Grouped parameters with and without decay, to better support AdamW best practices:

def get_optimizer(model, lr, weight_decay):
    decay, no_decay = [], []
    for name, param in model.named_parameters():
        if "bias" in name or "bn" in name or "BatchNorm" in name:
            no_decay.append(param)
        else:
            decay.append(param)
    return torch.optim.AdamW([
        {'params': decay, 'weight_decay': weight_decay},
        {'params': no_decay, 'weight_decay': 0.0}
    ], lr=lr)

‚úÖ Excludes BatchNorm and bias terms from regularization ‚Äî improving convergence and reducing over-penalization of critical parameters.
